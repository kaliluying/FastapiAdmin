## 第一部分：课程总纲

### 1.1 项目一句话定位

> **AI 劳动仲裁辅助系统 = FastAPI 后端骨架 + Vue3 前端页面 + RAG 链路 + 劳动仲裁业务场景**，用一个拖欠工资案例贯穿咨询、证据整理和文书草稿生成。

### 1.2 课程总目标

通过 4 天 32 学时，让学员**从零到一完成一个企业级 AI 应用项目**，并能完整答辩。

| 维度 | 目标 |
|---|---|
| 工程能力 | 跑通骨架项目，能定位接口、服务、RAG 和会话保存入口 |
| AI 能力 | 理解检索、Prompt 构建、模型回答三段链路 |
| 产品能力 | 交付 4 个核心环节（咨询 / 分析 / 证据 / 文书） |
| 软技能 | 能演示、能解释、能回答追问 |

### 1.3 四大核心功能（产品视角）

| # | 功能 | 入口 | 用户价值 |
|---|---|---|---|
| F1 | **AI 智能咨询** | `/ai/chat/ai-chat` + `/ai/chat/ws` | 7×24h 免费法律问答 |
| F2 | **维权诉求分析 + 胜诉评估** | 设计接口 / Prompt 演示 | 一键生成案件分析报告 |
| F3 | **证据智能管理** | `/ai/knowledge/document/upload` + 设计扩展 | 自动识别证据类型、评估证据效力 |
| F4 | **仲裁文书自动生成** | 设计接口 / 模板演示 | 5 分钟生成标准仲裁申请书 |

### 1.4 技术栈一览

```
后端：Python 3.12 + FastAPI + SQLModel + Alembic + JWT
AI  ：LangChain + OpenAI 兼容模型（具体模型以 `.env` 和 `/ai/chat/model-config` 为准）
向量：ChromaDB + BM25 (jieba)
前端：Vue3 + Vite7 + Element Plus + Axios
部署：uv + uvicorn / gunicorn + Nginx
包管理：uv（强制）
```

### 1.5 4 天路线图

```
跑起来    有依据回答  业务闭环    交付答辩
 │         │          │          │
环境      检索       证据        登录保护
骨架      Prompt     文书        部署检查
AI 基础   流式对话   前端展示    答辩演示
```

### 1.6 当前骨架接口口径

课堂讲解必须先区分“真实接口”和“设计扩展接口”。真实接口是当前仓库已经提供、可以直接在 `/docs` 或前端 API 封装中找到的接口；设计扩展接口用于课堂讲解产品闭环、Prompt 设计和前端原型，不要说成当前骨架已经完整联调。

| 能力 | 当前口径 | 接口 / 文件 |
|---|---|---|
| AI 会话列表 | 真实接口 | `GET /ai/chat/list`，见 `frontend/web/src/api/module_ai/chat.ts` |
| 创建 AI 会话 | 真实接口 | `POST /ai/chat/create` |
| 非流式 AI 问答 | 真实接口 | `POST /ai/chat/ai-chat` |
| WebSocket AI 聊天 | 真实接口 | `WS /ai/chat/ws`，端口以 `SERVER_PORT` 为准 |
| 模型配置查看 | 真实接口 | `GET /ai/chat/model-config` |
| 知识库列表 / 创建 | 真实接口 | `GET /ai/knowledge/list`、`POST /ai/knowledge/create` |
| 知识库文档上传 | 真实接口 | `POST /ai/knowledge/document/upload` |
| 文档重建索引 | 真实接口 | `POST /ai/knowledge/document/{id}/reindex` |
| 检索测试 | 真实接口 | `POST /ai/knowledge/retrieval/test` |
| 维权诉求分析 | 设计扩展 | 可用 Prompt / mock / 前端原型演示，接口名可暂定 `/api/claim/analyses` |
| 胜诉评估 | 设计扩展 | 可用 Prompt / mock / 前端原型演示，接口名可暂定 `/api/claim/evaluations` |
| 案件证据 AI 分析 | 设计扩展 | 当前知识库上传可演示“文件进入 RAG”，案件证据效力分析需另接业务接口 |
| 仲裁文书生成 | 设计扩展 | 可用 Jinja2 模板演示 HTML 草稿，接口名可暂定 `/api/documents/arbitration-application` |

- **P0 准备**：OpenAI API Key、`.env` 配 `OPENAI_API_KEY`
- **P1 核心**：`test_langchain.py` 调 `ChatOpenAI.invoke()` 返回一句话
- **P2 测试**：换 3 个不同问题都能正常响应
- **P3 优化**：多轮对话（拼接 history）、流式输出（`streaming=True`）

#### F2 知识库上传与索引
- **P0 准备**：3 份以上法律 `.md` 文档到 `data/law_sources/`
- **P1 核心**：`POST /ai/knowledge/document/upload` 成功，完成解析和索引入库
- **P2 测试**：5 个法律问题检索出对应法条
- **P3 优化**：调整 `chunk_size=700` / `overlap=120`，观察召回率

#### F3 RAG 检索
- **P0 准备**：知识库数据入库，确认 `ChromaKnowledgeStore` 可用
- **P1 核心**：`ChromaKnowledgeRetriever.retrieve()` 返回与问题相关的上下文
- **P2 测试**：用劳动争议问题验证“无上下文”和“有上下文”的回答差异
- **P3 优化**：按来源、相关度、业务标签调整结果排序

#### F4 Prompt 模板
- **P0 准备**：F1 跑通
- **P1 核心**：`RagPromptBuilder.build()` 把问题、上下文和输出要求组装成 Prompt
- **P2 测试**：golden case 集（10+ 条）回归通过
- **P3 优化**：Pydantic `output_schema` 强约束输出

#### F5 AI 聊天
- **P0 准备**：F1 + F4
- **P1 核心**：`/ai/chat/ai-chat` 或 `/ai/chat/ws` 接收消息 → RAG 检索 → LLM 返回 → 推回客户端
- **P2 测试**：浏览器调试器看到逐字输出
- **P3 优化**：`InMemoryCaseMemoryStore` 维护会话，断连清理

#### F6 维权诉求分析（设计扩展）
- **P0 准备**：F4
- **P1 核心**：Prompt 或设计接口返回 `{rights_claim, case_summary}`
- **P2 测试**：5 个案情描述全部能正确提取诉求
- **P3 优化**：JSON Schema 强约束，添加长度限制

#### F7 成功率评估（设计扩展）
- **P0 准备**：F6
- **P1 核心**：Prompt 或设计接口返回胜诉概率 + 建议
- **P2 测试**：3 个真实案情，概率合理（不应极端）
- **P3 优化**：建议具体到"收集 X 证据"而非"收集证据"

#### F8 文件上传
- **P0 准备**：FastAPI `UploadFile`
- **P1 核心**：`/ai/knowledge/document/upload` 接收知识库文件；案件证据上传作为设计扩展接口
- **P2 测试**：PDF / JPG / XLSX 都能上传
- **P3 优化**：路径设计 `data/evidence_uploads/{user_id}/{evidence_id}/`

#### F9 文件解析
- **P0 准备**：使用当前依赖 `pypdf`、`openpyxl`；OCR 为选做，需另装工具
- **P1 核心**：`extract_text_from_pdf/excel/image` 三个函数
- **P2 测试**：3 种格式各 1 个样本，提取出文字
- **P3 优化**：OCR 中文识别率优化（图像预处理）

#### F10 后台异步任务
- **P0 准备**：F8
- **P1 核心**：`BackgroundTasks.add_task(process_evidence, evidence_id)`，立即返回
- **P2 测试**：上传 1 个文件，1 秒内拿到响应，3 秒后 status=completed
- **P3 优化**：Celery + Redis（如果任务超过 1 分钟）

#### F11 证据 AI 分析（设计扩展）
- **P0 准备**：F8 + F9 + F4
- **P1 核心**：设计 `analyze_evidence(type, content, context)` 返回 `{type_detected, effectiveness, key_info, suggestions}`
- **P2 测试**：上传劳动合同 → 识别为"高"效力 + 提取合同期限
- **P3 优化**：建议具体到"补充 X 条款"

#### F12 文书生成（设计扩展）
- **P0 准备**：装 `jinja2`，写 `templates/documents/arbitration_application.html`
- **P1 核心**：模板或设计接口渲染 HTML 草稿
- **P2 测试**：传入完整案情 → 生成包含 9 个章节的申请书
- **P3 优化**：支持多模板（欠薪 / 工伤 / 辞退）

#### F13 性能优化
- **P0 准备**：F0
- **P1 核心**：`GZipMiddleware(minimum_size=1000)` + 连接池 `pool_size=10, max_overflow=20`
- **P2 测试**：压测对比（优化前 50KB → 优化后 8KB）
- **P3 优化**：异步 ORM、Redis 缓存热点 prompt

#### F14 JWT 认证
- **P0 准备**：装 `pyjwt`
- **P1 核心**：`backend/app/core/security.py` 完整（`create_access_token` + `decode_access_token` + `get_current_user`）
- **P2 测试**：登录返回 token，受保护接口 401/403 测试
- **P3 优化**：refresh token、token 黑名单

#### F15 密码安全
- **P0 准备**：装 `密码哈希`
- **P1 核心**：`set_password_hash` + `verify_password`
- **P2 测试**：同密码不同哈希、错误密码拒绝
- **P3 优化**：密码强度校验（长度、复杂度）

#### F16 扩展了解：第三方登录
- **P0 准备**：理解授权码换身份信息的基本流程
- **P1 核心**：能画出“前端拿授权码 -> 后端换身份 -> 本系统签发 JWT”的链路
- **P2 测试**：用 mock code 解释流程，不要求接真实平台
- **P3 优化**：了解 unionid、账号绑定和登录风控

#### F17 扩展了解：接口限流
- **P0 准备**：F0
- **P1 核心**：`RateLimiter` 内存实现 + 中间件
- **P2 测试**：60 秒 30 次后返回 429
- **P3 优化**：滑动窗口、Redis 集群支持

#### F18 部署
- **P0 准备**：Linux 服务器
- **P1 核心**：Gunicorn 4 worker 启动
- **P2 测试**：压测 100 并发不挂
- **P3 优化**：Nginx 反代 + systemd 守护 + 日志切割

#### F19 项目答辩
- **P0 准备**：Day 3 末 PPT 初稿
- **P1 核心**：演示脚本 + 5min 演示视频
- **P2 测试**：模拟答辩 5min + 2min Q&A
- **P3 优化**：常见 7 个 Q&A 提前演练
