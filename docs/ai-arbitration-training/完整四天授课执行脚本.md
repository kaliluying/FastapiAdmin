# AI 劳动仲裁辅助系统四天实训完整授课执行脚本

> 用途：讲师投屏课件网页时的逐步执行脚本。  
> 依据：`courseware/index.html` 课件、四天演讲稿、当前仓库后端/前端骨架。  
> 讲课主线：带学生用 AI 从 0 到 1 实现一个能演示、能解释、能答辩的劳动仲裁辅助系统。

---

## 0. 课前准备

### 0.1 讲师机器准备

**动作**

1. 打开课件网页：

```powershell
cd <项目目录>
python -m http.server 8765 -d courseware
```

浏览器打开：

```text
http://127.0.0.1:8765/index.html
```

2. 打开骨架项目：

```powershell
cd <项目目录>/backend
```

3. 准备三个常用窗口：

- 浏览器 1：课件网页。
- 浏览器 2：后端 `/docs`。
- 编辑器：骨架项目。
- 终端：后端启动、测试、前端启动。

4. 准备统一演示案例：

```text
张三在某公司工作 2 年，每月工资 8000 元。
公司拖欠 3 个月工资，并口头通知张三不用再来上班。
张三手里有劳动合同、工资流水、微信聊天记录。
他想知道能主张哪些权益，需要准备哪些证据，能否生成仲裁申请书草稿。
```

### 0.2 开场固定话术

**讲师说**

> 这四天我们不做一个“看起来很完整但讲不清”的项目。  
> 我们只围绕一条线：用户提出劳动争议问题后，系统如何找到依据、生成回答、整理证据、输出文书，并且最后能把这条链路讲给评审听。  
> 你们可以使用 AI，但不能把判断交给 AI。AI 负责加速读代码、拆任务、写初稿、定位错误；最终解释、验证和取舍必须由你们完成。

**学生动作**

- 记录统一案例。
- 每组确定 1 个主讲人、1 个代码负责人、1 个演示负责人。

**验收**

- 每组能用 30 秒说出：系统帮助谁、解决什么问题、最终输出什么。

# 第一部分：从零实现主线脚本

> 用法：这部分贯穿四天。每天开始前先回到对应节点，让学生知道当天在完整项目里的位置。

## 2. 从零实现 1：先定规则，人指挥，AI 协作

**网页位置**

- 顶部 Tab：`从零实现`
- 页面：`先定规则：人指挥，AI 协作`

**屏幕动作**

1. 展示课件页。
2. 打开 AI 工具。
3. 输入一条错误提示词：

```text
帮我实现一个劳动仲裁辅助系统。
```

4. 再输入一条正确提示词：

```text
你是我的结对工程师。先不要写代码，请帮我理解这个项目要解决什么业务问题，并列出最小可交付版本。
```

**讲师说**

> 第一条提示词的问题是范围太大，AI 会直接开始编，最后你们也不知道它为什么这么写。  
> 第二条提示词先让 AI 分析目标、边界和交付物。我们要训练的是工程协作，不是让 AI 替你做项目。  
> 从现在开始，任何代码生成前，都必须先问三件事：目标是什么，涉及哪些文件，怎么证明它能跑。

**学生动作**

- 每组写一条“坏提示词”。
- 再改成一条“可执行工程提示词”。

**验收**

- 学生能说出 AI 协作边界。
- 学生能把“大需求”拆成 3 到 5 个可验收小任务。

**救场**

> 如果学生只会问“帮我写代码”，立刻要求改成：“先不要写代码，请先分析项目结构和实现方案。”

## 3. 从零实现 2：从业务故事开始建项目

**屏幕动作**

1. 切到页面：`从业务故事开始建项目`。
2. 在白板或编辑器写用户故事：

```text
作为一名被拖欠工资的劳动者，
我希望输入自己的案情，
系统能给出初步咨询、诉求分析、证据建议和文书草稿，
以便我知道下一步怎么准备仲裁。
```

3. 把故事拆成四个模块：

```text
AI 咨询
诉求分析
证据管理
文书生成
```

**讲师说**

> 项目不是从技术栈开始的，而是从一个用户故事开始的。  
> 如果业务故事不清楚，后面 FastAPI、RAG、前端页面都会变成堆技术。  
> 第一版系统不追求功能多，只追求一条案例能从输入走到输出。

**学生动作**

- 每组用统一案例写 4 条用户故事。
- 每条用户故事写清：输入、处理、输出。

**验收**

- 每组有不超过 8 条的 MVP 用户故事。
- 每条故事都能对应一个页面、接口或结果。

## 4. 从零实现 3：让 AI 生成工程骨架

**屏幕动作**

1. 打开骨架项目目录：

```powershell
cd <项目目录>/backend
```

2. 展示骨架目录：

```text
app/api
app/plugin
app/core
app/config
app/scripts
main.py
```

3. 运行后端：

```powershell
uv sync
uv run main.py run --env=dev
```

4. 打开：

```text
http://127.0.0.1:<SERVER_PORT>/docs（当前骨架示例为 8004）
```

**讲师说**

> 骨架阶段只验证一件事：项目能不能跑起来。  
> 不要一开始就写数据库、RAG、文书生成。第一轮只要有启动入口、配置、路由和健康检查。  
> 这个骨架适合教学，因为它已经有 FastAPI 启动、路由组织、认证依赖、插件模块和测试。我们可以裁剪范围，把它当作课堂项目的底座。

**学生动作**

- 启动自己的后端。
- 打开 `/docs`。
- 找到健康检查或已有接口。

**验收**

- 后端能启动。
- `/docs` 能打开。
- 学生能说明 `main.py` 的作用。

**救场**

> 如果环境卡住，先不要讲复杂概念。按顺序检查：Python 版本、uv 是否可用、依赖是否同步、环境变量是否复制。

## 5. 从零实现 4：用 AI 设计数据模型和接口

**屏幕动作**

1. 切到页面：`用 AI 设计数据模型和接口`。
2. 展示接口契约表：

```text
POST /ai/chat/ai-chat        AI 咨询（真实接口）
POST /ai/knowledge/document/upload  知识库文档上传（真实接口）
POST /api/evidence/analyze   证据分析（设计扩展，当前骨架未内置）
POST /api/document/create    文书生成（设计扩展，当前骨架未内置）
GET  /ai/chat/list           会话历史（真实接口）
```

**讲师提醒**

> 当前仓库已经内置的是 AI 聊天和知识库接口。  
> 诉求分析、证据效力分析、文书生成属于课堂设计扩展，可以先用 Prompt、mock 数据或前端原型讲清业务闭环，不要说成骨架里已经有完整后端接口。

3. 打开骨架中的接口结构：

```text
backend/app/plugin/module_ai/chat/controller.py
backend/app/plugin/module_ai/chat/schema.py
backend/app/plugin/module_ai/chat/service.py
backend/app/plugin/module_ai/chat/crud.py
```

**讲师说**

> 接口是前端、后端和 AI 能力之间的合同。  
> 合同不清楚，后面就会变成“前端不知道传什么，后端不知道返回什么，AI 不知道生成什么”。  
> 所以我们先设计输入输出，再写表结构和 Service。

**学生动作**

- 选择一个用户故事。
- 写出接口路径、请求字段、响应字段、失败情况。

**验收**

- 每组至少完成 3 个接口契约：咨询、证据、文书。

## 6. 从零实现 5：实现 AI 咨询，先假数据，再接模型

**屏幕动作**

1. 切到页面：`实现 AI 咨询：先假数据，再接模型`。
2. 在接口文档里找到 AI 聊天接口。
3. 先演示 mock 返回：

```json
{
  "answer": "根据你的描述，可能涉及拖欠工资和违法解除。",
  "basis": ["劳动合同", "工资流水", "聊天记录"],
  "next_steps": ["整理证据", "计算欠薪金额", "准备仲裁申请"]
}
```

4. 再说明真实模型接入需要：

```text
OPENAI_API_KEY
OPENAI_BASE_URL
OPENAI_MODEL
```

**讲师说**

> 先 mock，不是偷懒，是工程顺序。  
> 如果第一步就依赖模型 Key、网络和第三方服务，课堂会被外部因素拖垮。  
> 我们先验证接口链路和前端展示，再把固定返回替换为真实模型。

**学生动作**

- 写一个 mock 咨询结果。
- 用接口或前端页面展示。

**验收**

- 无模型 Key 时，系统仍能返回结构化结果。
- 学生能解释后续如何替换成真实模型。

## 7. 从零实现 6：实现前端页面，先静态，再接接口

**屏幕动作**

1. 切到页面：`实现前端页面：先静态，再接接口`。
2. 展示前端骨架：

```powershell
cd <项目目录>/frontend/web
pnpm install
pnpm dev
```

3. 画出最小页面：

```text
AI 咨询页：输入框、提交按钮、回答区、依据区
证据管理页：上传区、证据列表、分析状态
文书预览页：案件信息、诉求、证据清单、文书草稿
```

4. 给 AI 的提示词：

```text
请为劳动仲裁辅助系统设计最小前端页面，只包含 AI 咨询、证据管理、文书预览三个页面，先给组件结构，不要接真实接口。
```

**讲师说**

> 前端第一版不要追求复杂样式。  
> 它要证明业务流程能被用户看见：用户输入问题，看到回答；上传证据，看到分析；生成文书，看到草稿。  
> 页面也按三步走：静态页面、mock 数据、真实接口。

**学生动作**

- 每组先做 AI 咨询页面。
- 至少实现四种状态：未输入、加载中、成功、失败。

**验收**

- 页面能显示 mock 结果。
- 失败状态有提示。
- 学生能说清哪一步替换为真实接口。

**救场**

> 如果学生卡在样式，先砍掉装饰，只保留输入框、按钮、结果区域和错误提示。

## 8. 从零实现 7：实现 RAG，知识库、检索、引用依据

**屏幕动作**

1. 切到页面：`实现 RAG：知识库、检索、引用依据`。
2. 打开：

```text
backend/app/plugin/module_ai/chat/rag.py
backend/app/plugin/module_ai/chat/service.py
backend/app/plugin/module_ai/chat/schema.py
backend/tests/test_ai_chat_service.py
```

3. 在白板画链路：

```text
用户问题 -> 检索上下文 -> 组装 Prompt -> 模型回答 -> 保存会话历史
```

4. 演示查询：

```text
公司拖欠我三个月工资，我可以申请劳动仲裁吗？
```

**讲师说**

> RAG 不是“把文档扔进向量库”这么简单。  
> 它的本质是让 AI 回答前先拿到依据。  
> 法律辅助场景最怕模型自由发挥，所以我们要让回答带着上下文、依据和边界。

**学生动作**

- 找到 RAG 入口。
- 写出三段链路：检索、Prompt、模型。

**验收**

- 学生能解释：为什么法律场景不能只问大模型。
- 学生能说明文件上下文如何进入回答。

**救场**

> 如果学生被类名绕住，停止讲类名，回到一句话：先找资料，再带资料问模型。

## 9. 从零实现 8：实现证据和文书，业务闭环

**屏幕动作**

1. 切到页面：`实现证据和文书：业务闭环`。
2. 展示证据材料：

```text
劳动合同：证明劳动关系
工资流水：证明工资标准和欠薪
聊天记录：证明催告和解除过程
```

3. 打开文件上传骨架：

```text
backend/app/api/v1/module_common/file/controller.py
backend/app/api/v1/module_common/file/service.py
```

4. 演示从证据到文书：

```text
上传证据 -> 解析内容 -> AI 分析证明目的 -> 填入文书模板 -> 生成草稿
```

**讲师说**

> 证据和文书决定这个项目是不是像真实业务系统。  
> AI 咨询只是入口，真正的业务闭环是把材料整理成可提交成果。  
> 但边界要讲清楚：AI 生成的是草稿，金额、日期、当事人信息必须人工复核。

**学生动作**

- 为三份证据分别写证明目的。
- 生成一段文书草稿。

**验收**

- 有证据清单。
- 有 AI 分析摘要。
- 有文书草稿。

## 10. 从零实现 9：让 AI 帮你调试，而不是猜错

**屏幕动作**

1. 切到页面：`让 AI 帮你调试，而不是猜错`。
2. 展示坏提示词：

```text
报错了，帮我修。
```

3. 展示好提示词：

```text
我在启动 FastAPI 项目时遇到下面错误。
目标：打开 /docs。
复现步骤：执行 uv run main.py run --env=dev。
报错全文：...
相关文件：main.py、app/config/setting.py。
请先分析可能原因，不要直接改代码；再给出验证顺序。
```

**讲师说**

> 报错不是拿给 AI 猜的，是给它诊断的。  
> 一条好的调试提示词必须包含目标、复现步骤、报错全文、相关文件、期望结果和已经尝试过的方法。  
> AI 连续两次修不好，就让它停下来分析原因，不要继续盲改。

**学生动作**

- 用自己的一个报错写调试提示词。
- 先让 AI 给原因和验证顺序，再决定是否改代码。

**验收**

- 学生能写出包含 5 个要素的调试提示词。

# 第二部分：四天逐日授课脚本

## 23. 常见追问标准回答

### 18.1 为什么不用纯大模型？

**回答**

> 因为法律辅助场景更看重依据和可控性。纯大模型可能回答得像专业文本，但无法保证依据来自我们的材料。RAG 的作用是先检索上下文，再让模型在上下文约束下回答。

### 18.2 系统能不能替代律师？

**回答**

> 不能。系统定位是辅助咨询、材料整理、风险提示和文书草稿生成。最终法律判断、金额核对、证据证明力判断都需要人工复核。

### 18.3 回答不准确怎么办？

**回答**

> 按四步排查：先看知识库材料是否正确，再看检索结果是否相关，再看 Prompt 是否约束清楚，最后看人工复核流程是否到位。

### 18.4 项目最大的技术亮点是什么？

**回答**

> 不是单纯调用大模型，而是把用户问题、文件上下文、检索资料、Prompt 构建、模型回答和会话保存串成一条可解释链路。

### 18.5 后续怎么扩展？

**回答**

> 可以扩展三方面：更完整的劳动法知识库、更细的证据类型识别、更规范的文书模板和人工复核流程。
