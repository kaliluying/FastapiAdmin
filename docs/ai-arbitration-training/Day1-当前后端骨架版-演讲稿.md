# Day 1 演讲稿：后端骨架启动与 AI 知识库最小闭环

> 适用场景：第一天授课。  
> 授课目标：让同学把项目跑起来，理解当前后端骨架真实具备的能力，并完成一次“知识库上传 -> 检索测试 -> AI 问答”的最小闭环。  
> 讲课原则：第一天不讲大而全，不提前展开证据分析、文书生成、第三方登录、限流和生产部署；先让学生建立项目能跑、接口能看、链路能懂的信心。

## 开场：今天我们先把系统跑起来

各位同学，今天是实训第一天。

这四天我们要做的不是一个简单的网页，也不是只调用一次大模型接口，而是一个完整的 AI 劳动仲裁辅助系统。

它最终要解决的问题是：一个普通劳动者遇到拖欠工资、违法解除、未签合同、加班费争议时，系统可以帮助他做初步咨询、整理证据、分析诉求，并生成仲裁申请书草稿。

但是第一天，我们不急着讲所有功能。

今天只有一个核心目标：**把当前后端骨架跑起来，并理解 AI 知识库的最小闭环。**

所谓最小闭环，就是：

```text
创建知识库
-> 上传文档
-> 后端解析文档
-> 文本切块
-> 生成向量
-> 写入 Chroma
-> 检索测试
-> AI 根据知识库回答问题
```

如果今天大家能把这条链路跑通，后面第二天讲 RAG 检索和 Prompt，第三天讲证据和文书，第四天讲安全、联调和答辩，都会顺很多。

## 为什么我们不从零开始

这里先说明一个问题：为什么这次实训不是从空文件夹开始写一个 FastAPI 项目，而是直接使用后端骨架？

因为真实企业开发里，大多数时候也不是从空文件夹开始写项目。

更常见的情况是：公司已经有一个后台系统，有用户体系、有权限、有数据库、有日志、有接口规范。我们要做的不是重新造一套后台，而是在这个已有系统里新增一个 AI 模块。

所以这次课程使用后端骨架，是为了让大家更早接触真实开发方式。

我们第一天不把时间主要花在创建目录、配置路由、搭登录、搭数据库这些重复基础工作上，而是直接学习：

```text
怎么看懂已有项目
怎么找到配置入口
怎么找到接口入口
怎么判断功能是否真实可用
怎么把 AI 知识库能力接到现有后端里
```

一句话总结：

```text
从零开始适合学框架入门；基于后端骨架开发，更接近真实企业项目。
```

## 一、先讲清楚这个项目是什么

我们这个项目叫 AI 劳动仲裁辅助系统。

它不是替代律师，也不是替代仲裁员。它的定位是辅助。

它主要辅助三件事：

第一，帮助用户理解自己的劳动争议问题。

比如用户问：

```text
公司拖欠我两个月工资，我应该怎么办？
```

系统可以结合劳动法知识库，给出初步说明、可能的仲裁请求、需要准备的证据。

第二，帮助用户整理材料。

比如劳动合同、工资流水、解除通知、聊天记录，这些材料后续都可以进入证据管理和知识库链路。

第三，帮助生成文书草稿。

比如仲裁申请书、证据清单、事实与理由说明。

但是注意，这些后面的业务能力不是第一天重点。今天我们先看当前后端已经具备的真实基础能力。

## 二、当前后端骨架已经有什么

现在大家打开项目目录，我们先不要急着写代码，先看骨架。

当前后端核心目录在：

```text
backend/
```

我们重点看几个地方。

这里放的是服务端口、数据库、Redis、JWT、OpenAI 配置、embedding 配置、Chroma 配置。

```text
backend/app/plugin/module_ai/chat/
```

这里负责 AI 对话、会话记录、RAG 组装、WebSocket 流式对话。

第三个是知识库模块：

```text

```

这里负责知识库创建、文档上传、文本解析、文本切块、embedding、Chroma 写入、检索测试。

第四个是启动入口：

```text
backend/main.py
```

我们启动后端时，就是通过这个入口运行。

所以当前项目不是空架子，它已经有一条 AI 知识库和对话的真实骨架。

## 三、讲环境：为什么要先把项目跑起来

很多同学做项目容易一上来就问：我要写哪个页面？我要调哪个模型？

但企业项目第一步不是写功能，而是确认项目能不能稳定启动。

一个后端项目要跑起来，至少需要四件事：

第一，依赖正确。

第二，配置正确。

第三，数据库和 Redis 能连上。

第四，接口文档能打开。

我们今天用 uv 管理 Python 依赖。

大家进入后端目录：

```powershell
cd backend
```

同步依赖：

```powershell
uv sync
```

然后启动项目：

```powershell
uv run main.py run --env=dev
```

这里我强调一下，不要只背命令。

`uv sync` 的意思是根据 `pyproject.toml` 和锁文件同步依赖。

`uv run` 的意思是在当前项目虚拟环境里执行命令。

`main.py run --env=dev` 的意思是使用开发环境配置启动后端。

如果启动失败，优先看三类问题：

第一，依赖没有安装好。

第二，`.env.dev` 配置不正确。

第三，MySQL 或 Redis 没启动，或者账号密码不匹配。

## 四、讲配置：配置文件决定服务怎么运行

接下来我们看配置。

当前后端主要配置在：

```text
backend/app/config/setting.py
```

开发环境变量模板在：

```text
backend/env/.env.dev.example
```

实际开发环境一般使用：

```text
backend/env/.env.dev
```

第一类配置是服务配置。

比如：

```text
SERVER_HOST
SERVER_PORT
ROOT_PATH
DOCS_URL
```

这些决定服务监听在哪个地址、哪个端口、接口前缀是什么、Swagger 文档地址是什么。

第二类配置是数据库。

```text
DATABASE_TYPE
DATABASE_HOST
DATABASE_PORT
DATABASE_USER
DATABASE_PASSWORD
DATABASE_NAME
```

数据库保存用户、角色、菜单、AI 会话、知识库、文档、chunk 元数据等结构化数据。

第三类配置是 Redis。

```text
REDIS_ENABLE
REDIS_HOST
REDIS_PORT
REDIS_DB_NAME
```

Redis 通常用于缓存、登录状态、临时数据等。

第四类配置是 AI 模型。

```text
OPENAI_BASE_URL
OPENAI_API_KEY
OPENAI_MODEL
EMBEDDING_PROVIDER
LOCAL_EMBEDDING_MODEL
```

当前 embedding 默认走本地小模型：

```text
BAAI/bge-small-zh-v1.5
```

这意味着文档向量化不需要每次都调用远程 embedding API。

第五类配置是 Chroma。

```text
CHROMA_PERSIST_DIR
CHROMA_COLLECTION_NAME
```

Chroma 用来保存文本向量和检索数据。

大家记住一句话：**MySQL 保存业务记录，Chroma 保存向量索引。**

## 五、讲接口文档：后端不是靠猜，是靠接口验证

项目启动后，我们要打开接口文档。

接口文档一般是：

```text
/docs
```

如果后端端口是 8004，那么常见地址类似：

```text
http://127.0.0.1:8004/api/v1/docs
```

具体以前端或后端配置为准。

打开接口文档以后，我们第一天重点看 AI 相关接口。

AI 会话相关：

```text
GET /ai/chat/list
POST /ai/chat/create
POST /ai/chat/ai-chat
GET /ai/chat/detail/{session_id}
GET /ai/chat/model-config
WS /ai/chat/ws
```

知识库相关：

```text
GET /ai/knowledge/list
POST /ai/knowledge/create
POST /ai/knowledge/document/upload
POST /ai/knowledge/document/{id}/reindex
POST /ai/knowledge/retrieval/test
```

这里要告诉同学一个工程习惯：**以后判断一个功能是不是真的存在，不要只看页面按钮，要看后端接口、数据库记录和实际链路。**

页面上有入口，不代表功能完整。

接口能调用、数据能落库、状态能更新、异常能处理，才算真正可用。

## 六、讲 AI 基础：大模型不是知识库

接下来讲 AI 基础。

大模型可以理解和生成文本，但它不是我们项目自己的数据库。

比如我们问模型：

```text
公司拖欠工资怎么办？
```

模型可能能给出通用回答。

但如果我们有自己的劳动法资料、案例材料、公司制度、证据材料，就不能只靠模型记忆。

我们需要先检索自己的资料，再让模型基于资料回答。

这就是 RAG。

RAG 的完整名字是 Retrieval-Augmented Generation，检索增强生成。

用一句话讲：

```text
先查资料，再回答。
```

更完整一点：

```text
用户问题
-> 把问题转成向量
-> 去知识库里找相关文本
-> 把相关文本和用户问题放进 Prompt
-> 调用大模型生成回答
-> 返回答案和依据
```

这里有几个关键概念。

LLM 是大语言模型，负责生成回答。

Prompt 是任务说明，告诉模型应该扮演什么角色、根据什么资料、输出什么格式。

Embedding 是把文本转换成数字向量。

Chroma 是向量库，保存向量和文本片段。

RAG 是把检索和生成串起来的一整条链路。

法律场景里为什么一定要讲 RAG？

因为法律问答最怕模型胡编。

所以我们的原则是：

```text
没有依据，不要乱答。
资料不足，提示补充。
结论重要，人工复核。
```

## 七、讲知识库闭环：今天最重要的实操

下面进入今天最重要的部分：知识库闭环。

当前后端已经有知识库模块。

我们要让同学理解一份文档从上传到被 AI 使用，中间经历了什么。

第一步，创建知识库。

知识库可以理解为一个资料集合。

比如我们创建一个：

```text
劳动法基础知识库
```

第二步，上传文档。

可以上传劳动合同法条、工资争议说明、仲裁流程说明等 `.txt` 或 `.md` 文档。

第三步，后端保存文件。

文件本身会被保存，数据库里也会记录文档信息。

第四步，解析文本。

后端会从文档中提取文字。

第五步，文本切块。

为什么要切块？

因为一整篇文档太长，模型和向量库都不适合一次处理全部内容。

切块以后，每一小段都可以单独向量化和检索。

第六步，生成 embedding。

当前项目默认使用本地小模型：

```text
BAAI/bge-small-zh-v1.5
```

它会把每个文本块转换成向量。

第七步，写入 Chroma。

Chroma 负责保存这些向量，后面用户提问时可以快速找相似内容。

第八步，检索测试。

比如我们问：

```text
公司拖欠工资可以申请劳动仲裁吗？
```

如果检索结果能返回相关法律资料，说明知识库索引是有效的。

第九步，AI 对话结合知识库回答。

最终，AI 不是凭空回答，而是结合检索出来的资料回答。

## 八、课堂演示流程

现在我建议按照这个顺序演示。

第一步，启动后端。

```powershell
cd backend
uv run main.py run --env=dev
```

第二步，打开接口文档。

确认 `/docs` 能访问。

第三步，查看模型配置。

重点看当前 chat 模型配置和 embedding 配置。

告诉学生：

```text
聊天模型负责回答；
embedding 模型负责把文档变成向量；
两者不是同一个概念。
```

第四步，创建知识库。

创建一个劳动法知识库。

第五步，上传一份简单文档。

建议第一天不要上传复杂 PDF，先用 `.txt` 或 `.md`。

内容可以是：

```text
用人单位拖欠劳动报酬，劳动者可以依法申请劳动仲裁。
劳动者应准备劳动合同、工资流水、考勤记录、聊天记录等证据。
```

第六步，观察索引状态。

告诉学生：上传成功不等于索引成功。

真正要看文档是否解析成功、chunk 是否生成、embedding 是否成功、Chroma 是否写入。

第七步，做检索测试。

问题可以是：

```text
公司拖欠工资，我应该准备什么证据？
```

如果能检索到刚才上传文档中的内容，就说明知识库链路跑通。

第八步，进入 AI 对话。

选择知识库，问同样的问题。

观察回答是否引用或结合了知识库内容。

## 十、课堂互动问题

这里可以问同学几个问题。

第一个问题：

如果大模型已经很聪明，为什么还要知识库？

参考回答：

因为大模型不知道我们自己的资料，也可能编造内容。知识库可以提供可控、可追溯的依据。

第二个问题：

上传文档成功，是否等于 AI 一定能用这份文档回答？

参考回答：

不一定。还要看文本是否解析成功、是否切块、是否生成 embedding、是否写入 Chroma、检索是否能召回。

第三个问题：

MySQL 和 Chroma 分别保存什么？

参考回答：

MySQL 保存知识库、文档、会话等业务记录；Chroma 保存文本块向量和检索数据。

第四个问题：

Embedding 模型和聊天模型有什么区别？

参考回答：

Embedding 模型负责把文本转成向量，用于检索；聊天模型负责根据 Prompt 生成自然语言回答。

第五个问题：

法律 AI 系统为什么要人工复核？

参考回答：

因为 AI 可能理解错误、资料不足或生成不严谨内容，法律结论不能完全自动化。
