# 四天实训 AI 提示词库

> 用途：讲师和学生在四天实训中使用 AI 编程助手、AI 问答模型、AI 文档助手时的统一提示词。  
> 口径：当前已经有后端骨架，AI 生成的前端界面是独立业务前台 / 演示前台，不嵌入后端管理平台；真实联调必须对齐后端接口，不得凭空创造接口、字段或假数据闭环。  
> 范围：开发提示词、业务 Prompt、联调提示词、验收提示词、答辩提示词。

---

## 1. 使用原则

### 1.1 前端是独立业务前台，但必须对齐后端接口

本实训中的前端界面由 AI 生成，定位是面向业务演示和学生实操的独立前台，不是后端管理平台里的一个菜单页。后端管理平台负责提供登录、权限、数据库、AI、知识库等基础能力；AI 生成前台负责承载劳动仲裁业务流程。

前端可以先生成界面原型，但进入联调时必须按下面顺序推进：

```text
确认后端接口
  -> 确认请求字段和响应字段
  -> 写前端 API 封装
  -> 写页面状态
  -> 写页面交互
  -> 联调接口
  -> 最后做视觉和体验优化
```

课堂统一要求：

- AI 生成前台必须和后端管理平台分开，不复用管理后台的信息架构、菜单结构和页面风格。
- 已有后端接口，前端必须调用真实接口。
- 没有后端接口，前端只能做“待接入”或 mock 演示，必须明确标注。
- 视觉原型可以使用示例数据，但不能把示例数据说成真实联调结果。
- 不允许为了页面好看而编造接口路径。
- 不允许在页面里散写 URL，优先使用 `src/api/...` 的接口封装。
- 不允许把真实 API Key、真实用户材料写进提示词或代码。

### 1.2 当前 AI 模块接口基线

当前前端已有 AI 模块接口封装：

```text
frontend/web/src/api/module_ai/chat.ts
frontend/web/src/api/module_ai/knowledge.ts
```

核心接口：

| 模块 | 方法 | 路径 | 用途 |
|---|---|---|---|
| 聊天 | GET | `/ai/chat/list` | 查询会话列表 |
| 聊天 | POST | `/ai/chat/create` | 创建会话 |
| 聊天 | PUT | `/ai/chat/update/{session_id}` | 更新会话标题 |
| 聊天 | DELETE | `/ai/chat/delete` | 删除会话 |
| 聊天 | POST | `/ai/chat/ai-chat` | 非流式 AI 对话 |
| 聊天 | GET | `/ai/chat/detail/{session_id}` | 会话详情 |
| 聊天 | GET | `/ai/chat/model-config` | AI 配置状态 |
| 聊天 | WebSocket | `/ai/chat/ws` | 流式对话 |
| 知识库 | GET | `/ai/knowledge/list` | 知识库列表 |
| 知识库 | POST | `/ai/knowledge/create` | 创建知识库 |
| 知识库 | PUT | `/ai/knowledge/update/{id}` | 更新知识库 |
| 知识库 | DELETE | `/ai/knowledge/delete` | 删除知识库 |
| 知识库 | GET | `/ai/knowledge/optionselect` | 启用知识库选项 |
| 文档 | GET | `/ai/knowledge/document/list` | 文档列表 |
| 文档 | POST | `/ai/knowledge/document/upload` | 上传文档 |
| 文档 | POST | `/ai/knowledge/document/{id}/reindex` | 重建索引 |
| 文档 | DELETE | `/ai/knowledge/document/delete` | 删除文档 |
| 检索 | POST | `/ai/knowledge/retrieval/test` | 检索测试 |

常用请求字段：

```ts
// AI 对话
{
  message: string;
  session_id?: string | null;
  knowledge_base_ids?: number[];
}

// 创建知识库
{
  name: string;
  description?: string | null;
  is_enabled: boolean;
  owner_dept_id?: number | null;
}

// 检索测试
{
  query: string;
  knowledge_base_ids: number[];
  top_k: number;
}
```

## 2. 通用提示词

### 2.1 接口对齐总提示词

```text
你是资深全栈工程师，负责把 AI 生成的独立业务前台与已有 FastAPI 后端骨架做接口对齐。

当前原则：
1. 前端界面是独立的 AI 劳动仲裁业务前台 / 演示前台，不嵌入后端管理平台。
2. 后端管理平台只作为登录、权限、数据库、AI、知识库等能力支撑。
3. 先读取后端接口、前端 API 封装和类型定义，再设计页面联调方案。
4. 不得凭空创造接口路径、请求字段或响应字段。
5. 前端必须优先使用已有 request 封装和 src/api 下的模块。
6. 如果某个功能后端接口不存在，必须明确标注“后端暂未提供”，不要用假接口冒充已完成。
7. 不修改后端骨架代码，除非讲师明确要求。
8. 视觉原型可以使用示例数据，但必须和真实联调状态区分。

请输出：
- 后端接口清单
- 独立业务前台页面清单
- 每个页面调用哪些接口
- 每个接口的请求字段和响应字段
- 页面 loading、空状态、错误状态
- 联调验收清单
```

### 2.2 前端页面生成总提示词

```text
你是 Vue3 + TypeScript + Element Plus 前端工程师。
请生成一个独立的「AI 劳动仲裁辅助系统」业务前台界面，不要做成后端管理平台菜单页。

界面定位：
- 这是面向业务演示和学生实操的独立前台。
- 后端管理平台只负责提供登录、权限、数据库、AI 对话、知识库和文档接口。
- 页面主线是案件驱动流程：智能咨询 -> 证据上传 -> 证据分析 -> 生成仲裁申请书。
- 视觉风格参考企业级 SaaS 工作台，专业、清爽、可信赖，不沿用通用后台管理页布局。

要求：
1. 使用 Composition API。
2. 使用已有 request 封装，不直接写 axios。
3. 使用已有 src/api/module_ai/chat.ts 和 src/api/module_ai/knowledge.ts。
4. 页面状态必须包含 loading、error、empty、success。
5. 表单字段必须与后端 schema 对齐。
6. 不新增后端接口。
7. 不使用真实密钥和真实个人信息。
8. 页面文案面向劳动仲裁辅助系统，不写通用后台占位文案。
9. 后端暂无接口的证据分析、文书生成等功能，只能做“待接后端”的设计态或 mock 演示态。
10. 视觉原型数据必须标注为示例数据，不得表述为真实联调成功。

请先输出接口对齐表，再输出实现方案。
```

### 2.3 业务 Prompt 总约束

```text
你是劳动仲裁辅助系统中的 AI 助手。
你只能提供信息整理、依据提示、风险提示和文书草稿辅助，不能替代律师、仲裁员或最终法律判断。

通用要求：
1. 不编造用户未提供的事实。
2. 不编造法律依据。
3. 不把 AI 输出表述为最终法律意见。
4. 事实不足时明确写“需补充”。
5. 回答必须区分“已知事实”“依据”“建议”“风险”。
6. 涉及金额、日期、主体信息时必须谨慎。
```


该案例可覆盖：

- Day1：AI 基础和知识库检索。
- Day2：RAG 问答和 Prompt 约束。
- Day3：证据链和文书草稿。
- Day4：权限边界、联调演示和答辩。

---

## 10. 讲师使用建议

1. 每次让 AI 生成前端前，先明确“独立业务前台，不嵌入后端管理平台”。
2. 先用“前端页面生成总提示词”生成业务前台视觉原型，再用“接口对齐总提示词”检查真实联调边界。
3. 每次让 AI 生成业务回答前，先使用“业务 Prompt 总约束”。
4. 每天结束用对应的“验收提示词”检查学生结果。
5. Day3、Day4 涉及后端可能暂未提供的证据/文书接口时，必须先查接口，不允许让前端伪装完成。
6. 答辩时优先展示真实跑通链路：登录、知识库、上传、检索、聊天。
7. 任何 AI 输出都必须保留人工复核提示。
