## 项目概览

AI 劳动仲裁辅助系统 是一个**全栈 AI 应用**，核心是 RAG（检索增强生成）驱动的劳动法律问答：

- **后端**：FastAPI 异步 + SQLModel + Alembic + JWT
- **AI 引擎**：LangChain + OpenAI Embedding（text-embedding-3-small，1536 维）+ ChromaDB
- **检索**：向量相似度 + BM25 关键词 + 多维结果排序 三路融合
- **对话**：WebSocket 流式 + 案件要素多轮补全
- **前端**：Vue3 + Vite7 + Element Plus（`static/` 为构建产物）

最核心的代码集中在：
- `backend/app/plugin/module_ai/chat/rag.py` — 检索引擎
- `backend/app/plugin/module_ai/chat/rag.py` — 对话与要素提取
- `backend/app/plugin/module_ai/chat/rag.py` — LLM / Embedding 封装
- `utils/prompts/` — 提示词工程（4 个文件）

## 第一档：项目命门（占工作量 70%+）

### 1. RAG 检索质量

**为什么是命门**：法律问答是"零容错"场景——AI 编造一条不存在的法条，用户拿去仲裁就会出问题。

#### 1.1 检索边界
- 同时跑 **ChromaDB 向量相似度 + BM25 关键词 + 多维结果排序**，三路结果加权融合
- 不同查询类型需要不同策略：
  - 法条精确查询（如"第 47 条怎么规定的？"）→ 靠 BM25 抓 `**第47条**` 这种显式引用
  - 语义查询（如"公司辞退孕妇合法吗？"）→ 靠向量检索
  - 关键词密集型查询 → BM25 分数更高

#### 1.2 法条引用解析
- `ARTICLE_PATTERN` 正则需同时识别阿拉伯数字与中文大写：
  ```python
  ARTICLE_PATTERN = re.compile(r"第\s*([一二三四五六七八九十百千万零〇两\d]{1,8})\s*条")
  ```
- `STRONG_REFERENCE_PATTERN` 抓 Markdown 加粗的 `**第X条**`
- 中文数字"四十七条"和阿拉伯数字"第47条"必须等价匹配

#### 1.3 多维结果排序
`_score_vector_candidate` 内的关键加分维度（`backend/app/config/setting.py` 中可配）：

| 维度 | 作用 |
|---|---|
| `RERANK_SOURCE_MATCH` | 用户指定某部法律时（如"按劳动合同法"）加分 |
| `RERANK_ARTICLE_MATCH` | 查询中的法条与文档标签匹配时加分 |
| `RERANK_CHAPTER_MATCH` | 章节命中加分 |
| `RERANK_KEYWORD_MATCH` | 关键命中加分 |

> 全部是经验值，调一组下来需要数小时 A/B 测。

#### 1.4 调参建议
1. 先建一个 **golden case 集**（20+ 条典型问题 + 期望答案）
2. 先检查 chunk 切分对不对（法条是否被切断）
3. 再调 RERANK 权重
4. 最后才考虑换 embedding 模型

### 2. 案件要素提取（多轮对话状态机）

**为什么难**：用户输入是模糊的自然语言，AI 必须从中抠出 8 个结构化字段：

```
employment_start / employment_end / dismissal_date / work_location /
company_location / has_labor_contract / salary_monthly /
company_action_type / current_evidence / claim_goal
```

#### 2.1 多轮补全
- `REQUIRED_CASE_FIELDS` 锁死必填项
- `FOLLOW_UP_QUESTIONS` 提供追问模板
- 用户先说一句"老板把我开了"，后面才说工资——必须等齐所有字段才能生成评估

#### 2.2 三个最磨人的正则
- **日期**：`DATE_PATTERN` 兼容 "2023.5"、"23年5月"、"去年"、"2023-05-01"
- **工资**：`SALARY_PATTERN` 兼容 "8k"、"8000元"、"八千块"、"月薪1.2万"
- **地点**：`LOCATION_PATTERN` 覆盖主要城市（北上广深杭苏成武西宁津渝长郑青甬锡佛东厦）

#### 2.3 意图分类（关键词映射）
`ACTION_TYPE_HINTS` 把口语映射到标准类型：
```python
"辞退" / "开除"          → 辞退
"解除劳动合同"            → 解除劳动合同
"协商解除"                → 协商解除
"拖欠工资" / "欠薪"       → 拖欠工资
"未签合同" / "没签合同"   → 未签劳动合同
```

#### 2.4 会话状态管理
- `InMemoryCaseMemoryStore` 用 `Lock` 保护多 WebSocket 并发
- session_id 维护 CaseProfile（事实 + 摘要 + 缺失字段 + 追问）
- **断线后状态仍在内存**，重启服务 = 所有用户案件全丢

### 3. 提示词工程

`utils/prompts/` 下四个文件把 prompt 当作"对象"组装：

| 文件 | 职责 |
|---|---|
| `prompt_types.py` | 类型定义 |
| `prompt_configs.py` | 各任务 prompt 模板 |
| `prompt_builder.py` | 动态组装 prompt |
| `output_schema.py` | Pydantic 输出 schema |

#### 3.1 多任务模板
不同任务 prompt 结构不同：
- **咨询**：系统角色 + 检索法条 + 用户问题
- **案件分析**：咨询 prompt + 案件事实
- **评估**：分析 prompt + 历史证据
- **文书生成**：评估 prompt + 模板结构

#### 3.2 输出 schema 强约束
- 用 Pydantic 锁死 JSON 结构
- 防止 LLM 返回 `{"result": "..."}` 这种半成品
- 前端可直接用 schema 反序列化

#### 3.3 幻觉抑制
- prompt 中必须明确写"只引用检索到的法条，不要补充未提供的法律"
- 必须配 **golden case 回归测试**，否则改 prompt 容易引入新幻觉

## 第二档：工程难点（会熬夜但不致命）

### 4. WebSocket 流式对话

- FastAPI 异步 + `AsyncGenerator` 流式输出
- 多 WebSocket 并发 + session_id 隔离
- 前端断连检测、服务端清理
- LLM 流式 token 通过 WebSocket 实时推给前端
- **坑**：异常处理要在 generator 内部 try/finally，否则会留下僵尸连接

### 5. 文本切片 Chunking

- 默认 `chunk_size=700` 字、`overlap=120` 字
- 必须"按段落 → 按句子"两级 fallback
- 不能从法条中间切断
- Markdown 标题、法条标记 `**第X条**` 要保留为 metadata
- **坑**：chunk 改一点点 → 对应文档需要重新索引 → `POST /ai/knowledge/document/{id}/reindex` 会重新消耗 Embedding 额度

### 6. ChromaDB 工程化

- 通过 `ChromaKnowledgeStore` 持久化向量索引
- 索引算法 HNSW（`hnsw:space=cosine`）
- 自定义 `ManualEmbeddingFunction` 防止 Chroma 偷偷用 ONNX 默认 embedding
- `_is_rebuildable_chroma_error` 处理 SQLite 崩溃后自动重建
- 关闭遥测：`CHROMA_TELEMETRY_ENABLED=False`
- **坑**：ChromaDB 1.x 与 0.4 持久化格式不兼容，升级时要 `--force` 重建

### 7. 异步 ORM 与数据库迁移

- SQLModel（基于 SQLAlchemy 2.0）异步 session
- Alembic 自动迁移 + 手动脚本（`fix_all_auto_increment.py` / `migrate_to_mysql.py`）
- SQLite / MySQL 双兼容（`utils/sqlite_compat.py` 强制升 SQLite 版本）
- **坑**：异步 session 必须在 `async with` 中使用，否则连接泄漏

## 第三档：软技能难点

| 难点 | 原因 |
|---|---|
| RAG 调参无标准答案 | 没有"准确率 100%"，只能人肉 A/B + 评估集 |
| 评估指标难定 | 答案好不好是主观的，需要人评 + LLM 自评双轨 |
| 法律知识随时更新 | 法条修订后要重建向量库，要写自动检测机制 |
| 前端改动困难 | `static/` 是 Vue3 **构建产物**，源码不在仓库根，需确认 `web/` 目录管理方式 |
| 幻觉责任 | AI 法律咨询是严肃场景，必须加免责声明 + 强制展示法条来源 |
| 团队协作 | RAG 调参、prompt 改动的"正确"主观性强，code review 标准难统一 |

## 三个最易低估的坑

### 坑 1：改 prompt 容易引入新幻觉
- **症状**：改了一行 prompt 看起来更好，结果发现某个 case 答错了
- **解法**：必须配 **golden case 回归测试**（20+ 条），改完跑一遍

### 坑 2：chunk 改一点点 → 整个索引重建
- **症状**：调 `chunk_size` 从 700 到 600，几百个 chunk 全部要重新 embedding
- **解法**：
  - 调 chunk 参数前先记录当前知识库、文档和检索测试结果
  - 重建时用 `POST /ai/knowledge/document/{id}/reindex`
  - 准备好 OpenAI API 余额（一次 embedding 几百个 chunk 也要几美分到几毛）

### 坑 3：WebSocket 断连后 case profile 还在内存
- **症状**：用户断网重连，发现对话接不上
- **解法**：
  - 实现 WebSocket 断连回调清理
  - 关键 case 数据要持久化到数据库（不能只放内存）
  - 服务重启前先 dump 一份
