# 二次开发工作流指南

## 方案：Fork + 自定义分支策略

### 第一步：初始化并关联上游

```bash
# 1. 在根目录初始化 git 仓库
git init

# 2. 添加上游远程仓库（替换为实际的原项目地址）
git remote add upstream <原项目的git地址>

# 3. 添加你自己的远程仓库（GitHub/GitLab/Gitee）
git remote add origin <你的仓库地址>

# 4. 拉取上游代码
git fetch upstream

# 5. 创建主分支并关联上游
git checkout -b main
git branch --set-upstream-to=upstream/main

# 6. 首次提交
git add .
git commit -m "feat: 初始化项目"
git push -u origin main
```

### 第二步：创建定制化分支

```bash
# 创建自己的开发分支
git checkout -b custom-dev

# 删除不需要的组件/文件
# ... 进行删除操作 ...

# 提交删除
git add .
git commit -m "chore: 移除不需要的组件"

# 推送到自己的仓库
git push -u origin custom-dev
```

### 第三步：同步上游更新

**方式 1：Rebase（推荐，保持提交历史整洁）**

```bash
# 1. 切换到主分支
git checkout main

# 2. 拉取上游最新代码
git fetch upstream
git merge upstream/main  # 或 git rebase upstream/main

# 3. 推送到你的远程仓库
git push origin main

# 4. 将上游更新合并到你的定制分支
git checkout custom-dev
git rebase main

# 5. 解决冲突（如果有）
# ... 解决冲突 ...
git add .
git rebase --continue

# 6. 推送更新（首次需要 force）
git push origin custom-dev --force-with-lease
```

**方式 2：Merge（更安全，保留完整历史）**

```bash
# 1-3 步同上

# 4. 将上游更新合并到定制分支
git checkout custom-dev
git merge main

# 5. 解决冲突
# ... 解决冲突 ...
git add .
git commit -m "merge: 同步上游更新"

# 6. 推送
git push origin custom-dev
```

### 分支策略

```
upstream/main  (上游主分支)
    ↓ fetch
origin/main    (你的主分支，纯净，与上游同步)
    ↓ merge/rebase
origin/custom-dev (你的定制开发分支)
```

### 日常工作流

1. **开发新功能**：在 `custom-dev` 分支上工作
2. **同步上游**：定期（如每周）执行"第三步"
3. **处理冲突**：如果你删除的组件被上游修改，选择保留删除状态

## 高级技巧

### 使用 .gitignore 排除不需要的目录

如果某些组件不想删除但也不想跟踪，可以添加到 `.gitignore`：

```gitignore
# 不需要的前端组件
frontend/src/views/module_application/
frontend/src/views/module_monitor/

# 不需要的后端模块
backend/app/api/v1/module_application/
```

### 使用 Git Submodules 管理独立模块

如果前端和后端需要独立版本控制：

```bash
# 将 backend 作为子模块
git submodule add <backend-repo> backend

# 将 frontend 作为子模块
git submodule add <frontend-repo> frontend
```

### 创建补丁文件保存定制化

```bash
# 生成你的定制化补丁
git diff main custom-dev > my-customizations.patch

# 在新克隆的仓库应用补丁
git apply my-customizations.patch
```

## 避免冲突的最佳实践

1. **模块化修改**：尽量在新文件中扩展，而不是修改原有文件
2. **配置分离**：将定制配置放在单独的文件中
3. **文档记录**：记录所有删除的组件和原因
4. **定期同步**：不要等太久才同步，避免大量冲突
5. **测试覆盖**：同步后立即运行测试，确保兼容性

## 删除组件检查清单

在删除组件前，检查：

- [ ] 是否有其他模块依赖该组件？
- [ ] 是否需要更新路由配置？
- [ ] 是否需要更新数据库迁移脚本？
- [ ] 是否需要更新 API 文档？
- [ ] 是否需要更新前端菜单配置？

## 常见问题

**Q: 如果上游修改了我删除的文件怎么办？**
A: Git 会提示冲突，选择保留删除状态即可（`git rm <file>`）

**Q: 如何只同步特定的上游更新？**
A: 使用 `git cherry-pick <commit-hash>` 选择性应用提交

**Q: 如何查看我的定制化内容？**
A: `git diff main..custom-dev` 查看所有差异
